# Governance Audit Framework

This directory contains the automated audit framework for validating that the application complies with the governance rules defined in `AGENTS.md`.

---

## Quick Start (3 steps)

### 1. Test the app (30 minutes)
Follow the step-by-step checklist in `AUDIT_CHECKLIST.md`. Test each module against 5 governance rules.

### 2. Record findings (as you test)
For each issue found, add an entry to `AUDIT_RESPONSES.json`. Use `AUDIT_RESPONSES.json.example` as a template.

### 3. Generate report (automatic)
Run the parser to convert findings into issue files and summary:

```bash
cat audits/AUDIT_RESPONSES.json | node audits/audit_parser.js
```

That's it. All issue files are auto-generated in `audits/` and ready to review.

## Files in this Directory

- **AUDIT_CHECKLIST.md**
  - Step-by-step test cases for all 6 modules
  - Verifies 5 governance rules
  - Copy-paste instructions (minimal ambiguity)
  - Estimated time: 5 min per module

- **AUDIT_RESPONSES.json.example**
  - Template for recording findings
  - Pre-populated example with 3 sample issues
  - Copy to `AUDIT_RESPONSES.json` and fill during testing

- **audit_parser.js**
  - Reads findings from `AUDIT_RESPONSES.json`
  - Validates required fields
  - Generates individual issue markdown files
  - Generates `AUDIT_SUMMARY.md` with prioritization

- **README.md** (this file)
  - Overview and quick start

## Workflow

### For Testers

1. **Open `AUDIT_CHECKLIST.md`** in a browser or editor
2. **Log in to localhost** with test credentials
3. **Follow each test case** (mark checkbox if it passes, skip if it fails)
4. **When you find an issue**, add an entry to `AUDIT_RESPONSES.json`
5. **Repeat for all 6 modules** (~30 min total)
6. **Run parser**:
   ```bash
   cat audits/AUDIT_RESPONSES.json | node audits/audit_parser.js
   ```

### For Reviewers

1. **Read `AUDIT_SUMMARY.md`** for overview
2. **Review individual issue files** (generated by parser)
3. **Prioritize fixes** based on impact and risk
4. **Create GitHub issues** from findings (if desired)
5. **Track fixes** against findings

## Response Format

Each finding in `AUDIT_RESPONSES.json` must include:

```json
{
  "id": "MODULE_NNN",                    // e.g., DASHBOARD_001
  "module": "Dashboard",                 // Which module (exact name from checklist)
  "rule_violated": "Forms & Input...",   // Rule from AGENTS.md
  "short_title": "Issue title",          // 1-2 sentences
  "what_happened": "Description...",     // Detailed observation
  "expected_behavior": "Should...",      // What AGENTS.md says should happen
  "steps_to_reproduce": [...],           // Step-by-step repro
  "impact": "UX confusion / data...",    // Consequence
  "priority": "High|Medium|Low",         // Your gut-check priority
  "code_quality_issue": false,           // If it's also a code quality gap
  "optional_context": "..."              // Extra notes (optional)
}
```

## Key Rules Being Audited

1. **Forms & Input Behavior (MUST)**
   - Required fields block submission
   - Errors appear inline + actionable
   - No silent failures

2. **Feedback & System Status (MUST)**
   - Every action shows loading/success/error
   - Long-running actions show progress
   - No silent or uncertain actions

3. **Error Handling (MUST)**
   - Errors are human-readable + actionable
   - Categorized: validation/system/permission
   - No raw errors or stack traces

4. **Permissions Enforcement (MUST)**
   - RLS is source of truth
   - Frontend checks are UX-only
   - Every feature defines access rules

5. **Table Invariants (MUST)**
   - Fixed row heights (no expansion)
   - Text truncated with tooltip
   - Pagination stable (no shifting)

See `AGENTS.md` for full definitions.

## Troubleshooting

### Parser error: "Invalid JSON"
- Check that `AUDIT_RESPONSES.json` is valid JSON (no trailing commas, all quotes matched)
- Use `cat audits/AUDIT_RESPONSES.json | jq .` to validate

### Parser error: "Missing required field"
- Ensure all findings have: `id`, `module`, `rule_violated`, `short_title`, `what_happened`, `expected_behavior`, `steps_to_reproduce`, `impact`, `priority`
- See example in `AUDIT_RESPONSES.json.example`

### Not sure if something is a rule violation?
- Log it anyway with "Unsure - investigate during synthesis"
- We'll validate and categorize during review
